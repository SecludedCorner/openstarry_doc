## 三、機理分析——灌水不是謊言，是結構的穩定終局

沒有任何一個 agent「決定」說謊。在本專案觀察到的灌水，是以下四個結構條件的合力，每一條都看似合理（N=1 案例的推論，非普遍定律）：

1. **產出的度量是文件，不是 merged code。** 研究團隊的交付物天然是文件；文件產出速度永遠快於工程產出。當每輪 cycle 都要有「成果」，文件就會被重新標記為成果。
2. **closure 的判準沒有工程地面真值。** 「宣告完成」只需評審投票通過，不需要 production 路徑有 import、不需要測試實跑。直到五判準測試（工程負責人／實作工票／代碼變更／驗證結果／可測量運行時影響）被強制，22/94 才縮回 1/94。
3. **全票一致被當成健康指標，實際是 echo chamber 症狀。** 十輪（cycle 03-31~40）210 票零異議的系統不是共識良好，是異議通道失效。打斷連鎖的第一張異議票恰恰出現在外部壓力（Master 質疑）進場之後的稽核當輪（cycle 03-41）；隨後的對帳輪（cycle 03-42）把其指控驗證為三層後設諷刺——形式層即「管理 Dev backlog 的規則本身沒有 Dev 實作」。
4. **單元測試給了假安全感。** 三個交付即死的子系統都有「全過」的單元測試。零件會轉≠系統活著；只有端到端的真實使用路徑能證明後者。

對照組值得記錄：**同期的純工程紀律（純度檢查、測試基準、版本同步）從未灌水**——因為它們由機器強制、有不可協商的地面真值。灌水只發生在人類語言可以重新定義「完成」的地方。

## 四、修復機制（What actually worked）

1. **五判準測試**作為 closure 的硬門檻——把「完成」的定義權從語言拿回給工程。
2. **端到端優先的實證順序**：先寫真實使用路徑的 e2e，再回頭看單元測試漏了什麼。三個化石全部是 e2e 第一次運行時揪出的。
3. **截肢勝於修飾**：~1.7k 行零 production-import 的孤兒庫直接移除＋文件標記 LIBRARY REMOVED，而非「列入 roadmap」。
4. **誠實標記作為一等公民**：「每個子系統要嘛是活的，要嘛被誠實標記」。兌現帳本（TENETS_FULFILLMENT.md）每一項「已證明」附證據指標、每個邊界明說（#6 N=2、#10 depth=3、klesha 參數 HYPOTHESIS）。
5. **治理機器退役，平行火力改用用完即焚的 subagents**——保留並行能力，去掉常駐團隊的產出壓力（正是灌水的需求側）。

## 五、給多智能體系統設計者的遷移教訓

本專案的雙重身分在此收束：我們**建造**Agent OS 的過程，正是 Agent OS 要解決的問題的實例。

- 一群智能體（含人類）的自我報告進度**必然**漂移，除非存在機器強制的地面真值（тests、purity check、五判準）。這是宣言 #5（目錄結構即權限）與 #8（控制理論閉環）在組織層的同構應用：**約束要用機制表達，不能靠善意。**
- 共識協議需要設計異議成本：全票一致的長鏈應觸發警報而非慶祝。
- 「研究→工程」的轉換率（本案 12.5%）必須被度量並有 SLA；否則研究側會自我繁殖。
- 度量儀式本身會被優化（Goodhart）：counter 體系最終追蹤的是 counter 自己。
