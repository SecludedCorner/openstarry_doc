## 4. Binding Text

### 4.1 §78.1 — Scope（By-Ratification-Status）

> TW 翻譯對等依文件實例之 *by-ratification-status* tier 分類為必要（或建議、或可選）：
>
> - **BINDING** tier（Tenets #1-#10、Master Rulings MR-1~MR-13、Zero Tolerance ZT-1/2/3、Baseline Rules #1-#78、ratified ENG-FAB items v1.X binding） — TW parity **MUST**。
> - **CANDIDATE** tier（canonical openstarry_doc/ ratified 子目錄、ratified Plan specs、ratified Master Ratification batches 吸收進 canonical） — TW parity **SHOULD**。
> - **KNOWLEDGE_ONLY** tier（canonical openstarry_doc/ Reference/ 研究筆記、calibration 報告、技術 spec 吸收後） — TW parity **SHOULD**（與 CANDIDATE 相同；差異於 §78.6 retroactive audit 排程）。
> - **CONSULTATIVE** tier（in-flight Plan specs、draft Master Ratification batches、R-stage drafts 升級供下輪參考） — TW parity **MAY**。
> - **SCRIBE-internal** tier（G0-G5 process gate logs、GP-coherence logs、R-stage 工作 artefacts、side-bar discussions、retrospectives、daily logs、deliver/ cycle-internal artefacts） — TW parity **MAY**。

Tier 分類為 **by-ratification-status**，非 by-location 亦非 by-doc-class enumeration（依 D-§2-01 17/4/2；DSS-A2 dissent 保留 §10）。給定文件之 tier 可能跨輪變動若 ratification status 改變（如 CANDIDATE → BINDING 經 Master Ratification；CONSULTATIVE → CANDIDATE 經 R4 close）；parity 需求對應更新。

### 4.2 §78.2 — Strength（Static + 時序升級）

> Strength 依 §78.1 tier 靜態分層（governance MUST / canonical SHOULD / Plan/operational MAY）依 D-§2-02（18/5；DSS-B1 5 dissent §10）。**時序升級**條款適用：以 EN-only 落地於 BINDING 或 CANDIDATE tier 之文件取得 **24 小時 grace window** 至 R3+1 cycle close 取得 TW parity；R3+1 cycle close 失敗 = TW parity **MUST**（governance）或 **SHOULD**（canonical）違反，依 §78.7 CI gate（D-§2-02 19/4；DSS-B1b 4 dissent §10）。

升級**自動化**經 F-15 v3 linter timestamp 比較（EN sibling mtime vs TW sibling presence）。無需手動 tracker。

### 4.3 §78.3 — Timing × Responsibility

> 依 scope tier（依 §78.1）共同 binding（依 D-§2-03 DEFER R4 absorb；UNANIMOUS）：
>
> - **BINDING tier**：same-PR-strict timing；primary author + SCRIBE/LINNAEUS reviewer（無 dual-author 依 D-§2-04 5/18）；MR-9 繼承 — 若 MUST 不能於 same-PR 滿足，文件不進 BINDING tier 直至 parity 達成。
> - **CANDIDATE / KNOWLEDGE_ONLY tier**：24h grace + R3+1 cycle close timing 依 §78.2 時序升級；primary author + SCRIBE post-audit 於 G4-folder gate。
> - **CONSULTATIVE tier**：cycle-end timing（R4 close）；primary author 盡力 + SCRIBE informational flag。
> - **SCRIBE-internal tier**：無 timing 約束（MAY）；primary author 可選。

### 4.4 §78.4 — Quality（3 子條款）

> 依 D-§2-04（3 子投票）：
>
> - **Glossary YES 配合 epistemic 前綴**（21/2）：每個 glossary entry **應** 攜帶 `verified:` / `inferred:` / `speculative:` 前綴 tag 依 ASANGA Yogācāra 知識論紀律。前綴隨社群使用沉澱詞義而更新。初始 v1 glossary 位於 `tools/tw_glossary.yaml` 依 O1 §6.1（~30 個初始 entries）。
> - **結構保真 layer-bound**（22/1）：TW 應保留 machine-checkable 結構（heading 階層、表格列數、code-block 內容、front-matter keys 相同 values 翻譯適度、cross-reference sibling-anchor 穩定）MUST；human-checkable 結構（bullet 順序、footnote anchor 穩定）SHOULD；un-checkable 層（句子層對齊、prose-paragraph parity）MAY（依 ASANGA + TURING 層分解）。
> - **Dual-author NO**（5/18；DSS-B2 5 dissent §10）：primary author 撰寫 TW；SCRIBE/LINNAEUS 於 G4-folder gate 擔任 reviewer 角色。Dual-author 起草**未強制**；review-after-draft 模式採用。

### 4.5 §78.5 — Retroactive Policy

> 依 D-§2-05（forward-only with scheduled audit per release-tag；16/4/3；DSS-B3 7 dissent §10）：
>
> - Rule #78 適用 **forward-only** 自 cycle 03-15 起。已存在於 cycle 03-15 之前落地之 canonical openstarry_doc/、Tenets、MR、ZT、Baseline Rules、Plan specs **祖父化 EN-only** 而**不**受回溯 TW parity 強制依 MR-12 既有不破壞。
> - **Scheduled audit per release-tag**：每 release-tag 邊界（依 Rule #75 §75.X pnpm build cadence），SCRIBE 對 canonical corpus BINDING + CANDIDATE tier 執行 TW-parity sweep 並 flag 任何缺 TW parity 的文件以求 owner 注意。Owner 可遵循（撰寫 TW 於 R3+1 cycle close grace 內）或記載 refusal-with-rationale（CONSULTATIVE 降級或腳註 per case）。
> - **Cycle 03-13 `plugin-gear-arbiters.md` backfill 分類** = "MR-10 retroactive precedent (one-off)" 依 D-§2-05 子投票（14/5/3/1；DSS-B3b 9 dissent §10）。Cycle 03-14 audit #93 補救被認定為 MR-10 retroactive precedent **僅對此單一文件**，非作為 standing retroactive policy。未來類似 gap 依 §78.5 forward-only + scheduled audit (b)。見 sibling `Reference/12_Cycle03-13_Backfill_Classification.md`。

### 4.6 §78.6 — CI Gate（Multi-Layer）

> 依 D-§2-06 UNANIMOUS multi-layer 規範：
>
> 1. **F-15 linter `tools/f15_check.py` 擴充** — 於 PR / commit 時間；檢查 sibling-file presence 依 §78.7 sibling-naming convention；per-tier 規則套用透過 config-driven dispatcher。失敗模式 = block-PR 對 BINDING tier；warning 對 CANDIDATE / KNOWLEDGE_ONLY tier；informational 對 CONSULTATIVE / SCRIBE-internal。
> 2. **pnpm build 整合** — 於 release-tag 邊界（Rule #75 §75.X 互動）；BINDING-tier TW parity gap 偵測則 fail build。Hard fail = block-release。
> 3. **Doc gate** — 於 release-tag 邊界；整合 F-15 linter output 進 release artefact；對齊 Rule #75 §75.X 位置 gate 語意。
> 4. **G4-folder gate** — 於 cycle R4 close（SCRIBE process gate）；G4-folder-7（cycle 03-15+ 新子檢查）驗證 cycle deliver/ 輸出之 TW parity *依其落地 tier*。
> 5. **Coordinator hook** — 於 Suggestion → Canonical 4-mirror sync（cycle 03-14 invariant）；coordinator subagent 執行 file-list diff 依 §78.7 sibling-naming convention 並 flag tier-violations。BINDING hard fail；CANDIDATE warning。

依 WIENER 閉環分析，single-loop coverage 留下 moment-of-failure gaps；multi-layer 跨所有時點維持閉環（PR / release-tag / cycle-end / mirror-sync）。Author-fatigue trade-off 經 UNANIMOUS R3 投票接受。

### 4.7 §78.7 — Sibling-Naming Convention

> 依 MRB-§2-02（TURING + SCRIBE 編碼化）。canonical sibling-naming pattern 為 `<basename>.<lang>.md`。範例：
>
> - `plugin-gear-arbiters.md`（canonical EN — 當語言由 project default 隱含；v0 / cycle 03-13-pre 形式）。
> - `plugin-gear-arbiters.en.md`（canonical EN — 當顯式 `.en.` 後綴使用；**對 Rule #78 forward-only 落地之新文件偏好**）。
> - `plugin-gear-arbiters.tw.md`（TW parity sibling）。
>
> `.en.` 後綴**建議但非強制**用以向 cycle 03-13-pre EN-only docs backward compatibility；規則依 EN base file 在任一 convention 下之 *`.tw.md` sibling 存在* 套用 parity check。未來 cycles（03-20+）可在觀察期後 bind `.en.` 後綴為 MUST（informational schedule 依 O1 §9.3）。
