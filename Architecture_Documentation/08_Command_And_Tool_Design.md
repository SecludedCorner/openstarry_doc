# 07. 命令 (Command) 的設計與定位

在我們的架構中，`Command` (命令) 是一個多層次的概念，根據其上下文的不同，它由系統的不同部分來處理。本文件旨在澄清 `Command` 與 `Tool` 的關係，以及它在架構中的具體定位。

---

## 1. 作為「用戶系統命令」 (在 UI 插件中)

這是最直接面向用戶的一種命令，用於控制代理人客戶端或系統本身，而非交由 LLM 處理的任務。

*   **定義：** 以特殊前綴（如 `/`）開頭的字符串，例如 `/help`, `/reset`, `/status`。
*   **處理者：** **UI 插件 (UI Plugin)**。
*   **工作流程：** `UI` 插件在接收到用戶輸入時，首先進行攔截判斷。如果識別出是系統命令，則由 `UI` 插件自身進行解析和處理，不進入「代理人核心」的對話流程。這種設計將用戶對「系統」的操作和對「代理人」的對話任務清晰地分開。

---

## 2. 作為「工具」的同義詞 (在工具插件中)

在許多語境下，`Command` 和 `Tool` 的概念可以互換。

*   **定義：** 一個可供 LLM 調用的、原子化的、具有單一職責的功能函數，例如 `read_file`。
*   **處理者：** **工具插件 (Tool Plugin)**，並由「代理人核心」的執行循環調度。
*   **結論：** 在我們的架構文檔中，為了保持術語的統一，我們將此類功能統一稱為 **`Tool`**。因此，如果 `Command` 是在這個意義上被使用，它就對應我們設計中的 `Tool`。

---

## 3. 作為「高階工作流命令」 (在代理協調層中)

這是最高級、最強大的一種命令，代表一個可被調用的、由多個步驟組成的複雜任務。

*   **定義：** 一個具名的、可複用的**「工作流圖 (Workflow Graph)」**。例如，一個名為 `generate_weekly_report` 的命令，其背後可能是一個包含數據查詢、圖表生成、LLM 總結和郵件發送等多個節點的工作流。
*   **處理者：** **代理協調層 (Agent Coordination Layer)**。
*   **工作流程：**
    1.  用戶、定時任務或外部 API 觸發一個高階命令。
    2.  「代理協調層」的**路由組件**或**命令分派器**接收到請求，並識別出命令的名稱。
    3.  協調層加載與該命令對應的預定義工作流圖。
    4.  協調層的**工作流引擎**負責執行該圖中的所有節點，完成整個複雜任務。

這種設計使得我們可以將複雜的業務邏輯封裝成一個個簡單的、可被調用的「命令」，極大地提升了系統的複用性和能力上限。

---

## 4. 專案級配置目錄 (.openstarry/)

`.openstarry/` 是一個特殊的專案級配置目錄，允許 Agent 在特定工作目錄中應用限制性的安全、功能及外掛設定，而不影響系統級別的全域設定。此機制實現了 Tenet #5「目錄結構即權限」的完整覆蓋——從系統層 (`~/.openstarry/`) 到專案層 (`./openstarry/`)。

### 三文件結構

`.openstarry/` 包含三個 JSON 檔案，各司其職：

| 檔案 | 用途 | 合併策略 |
|------|------|---------|
| **config.json** | 認知配置（温度、模型、token 預算、最大重試次數） | 專案覆寫系統 |
| **permissions.json** | 安全邊界（允許路徑、允許工具、被拒工具、token 天花板） | 交集 / 最小值（縮小權限） |
| **plugins.json** | 外掛清單（僅載入指定外掛，完全覆寫系統外掛清單） | 完全覆寫 |

### 限制性合併模型 (Restrict-Only)

專案級配置遵循一個核心原則：**專案配置只能縮小權限，永遠不能擴大權限**。

- **security ceiling** (安全天花板) — 取交集：`allowedPaths`、`allowedTools`、`maxConcurrentTools`、`maxTokens` 採用交集
- **security floor** (安全底線) — 更嚴格：`confidenceFloor`、`safetyMinimumGear` 採用最大值（更限制）
- **denied set** (拒絕集合) — 並集：`deniedTools` 採用並集（多一個拒絕工具）
- **neutral fields** (中立欄位) — 專案覆寫：`identity.name`、`cognition.temperature` 等專案特定值覆寫系統設定

### 十步驟驗證流程

`.openstarry/` 目錄的載入和驗證遵循以下流程：

1. **偵測** — `findProjectRoot()` 遞迴向上查找 `.openstarry/` 目錄
2. **讀取** — 若目錄存在，讀取三個 JSON 檔案
3. **路徑安全檢查** (SEC-003) — `isPathSafe()` 驗證候選路徑，防止符號連結攻擊 (realpathSync)
4. **Schema 驗證** — 使用 Zod `.strict()` schema 驗證 JSON 結構，拒絕未定義欄位
   - 若失敗 → `SecurityError` 擲出（停止啟動）
5. **權限合併** — 執行 restrict-only 合併邏輯
6. **欄位驗證** — 驗證合併後的欄位值（e.g. `allowedPaths` 必須非空）
7. **插件清單驗證** — 若 `plugins.json` 定義，驗證每個插件是否存在
8. **衝突檢測** — 檢測 `allowedTools` 與 `deniedTools` 的重疊
9. **審計記錄** — 記錄載入的專案配置內容
10. **失敗處理** — Steps 1-4 的失敗為 `SecurityError`（致命）；Steps 5-9 的失敗為 `ConfigError`（可降級或跳過）

若目錄不存在，系統跳過本流程，僅使用系統級配置。

### CLI 支援與旗標

- **`openstarry init --project`** — 在目前目錄產生 `.openstarry/` 範本及三個空範本檔案
- **`openstarry start --no-project-dir`** — 強制跳過專案級配置探測，僅使用系統級配置
- **`openstarry version --verbose`** — 顯示載入的系統級和專案級配置詳細內容

### 啟動時載入，不支援熱重載

專案級配置在 Agent 啟動時一次性載入，不支援執行時重載。此設計權衡考量：
- **優勢**：避免 TOCTOU（檢查-使用時間差）競速漏洞
- **限制**：若需變更 `.openstarry/` 配置，需重啟 Agent
