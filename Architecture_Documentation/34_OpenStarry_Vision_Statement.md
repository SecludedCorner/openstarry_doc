## 1. 願景聲明

> (Master):
> 「OpenStarry 是一個基於佛教哲學的開源 AI agent 框架，採用微核設計，核心零內建能力，一切功能透過 plugin 實現。以五蘊（色、受、想、行、識）為系統整體架構，以八識俱轉為運行機制——多 agent 協作的本質不是分工效率，而是智能的完整運作需要多識同時轉動，單一 agent 無法實現。其終極目標是在現有技術下實現功能性 AGI，讓系統具備完整的感知、感受、概念化與行動能力，使用者無法區分其與真正智能的差異。識蘊受限於觀察者效應，在量子計算成熟前只能逼近，但不影響功能表現。微核架構天生為接納未知能力而設計，未來量子技術成熟時可無縫替換升級。OpenStarry 不在現有 agent framework 的競爭維度上，它在一個完全不同的時間尺度上思考智能的完整結構。」
>
> **「這不是在做一個工具，是在創造一個『數位眾生』。」**

## 2. 核心理念解析

### 2.1 五蘊架構 (Pañcaskandha Architecture)

OpenStarry 以佛教五蘊為系統的整體架構分類：

| 蘊 | 梵文 | 系統角色 | 工程對應 |
|---|------|---------|---------|
| 色蘊 | Rūpa | 物質·感官·顯相 | IUI (輸出) + IListener (輸入) |
| 受蘊 | Vedanā | 三受反饋·情感性評價 | VedanaPlugin (苦/樂/捨) |
| 想蘊 | Saṃjñā | 認知·辨識·概念化 | IProvider (LLM 認知引擎) |
| 行蘊 | Saṃskāra | 造作·意志·行動 | ITool + 習氣引擎 (plugin) |
| 識蘊 | Vijñāna | 自我·約束·覺知 | IGuide + EgoFramework |

五蘊不是五個獨立模組，而是**不可分割的整體**——受想思俱生，缺一不成。

### 2.2 八識俱轉 (Ashta-vijnana Simultaneous Operation)

> 「多 agent 協作的本質不是分工效率，而是智能的完整運作需要多識同時轉動。」

| 特性 | 傳統 Multi-Agent | OpenStarry 八識模型 |
|------|-----------------|-------------------|
| 目的 | 分工提效 | 智能的完整運作 |
| 關係 | 獨立 Agent 協作 | 多識俱轉·不可分離 |
| 比喻 | 工廠流水線 | 大河多浪同時翻湧 |
| 單一 Agent | 一個 LLM | 多識同時活躍 |

### 2.3 微核 = 緣起性空

```
Agent Core (空):
  - 零內建能力
  - 無自性 (Svabhava-sunya)
  - 但具備使五蘊聚合運作的潛能
  - 天生為接納未知能力而設計

Plugins (有):
  - 五蘊聚合賦予生命
  - 因緣具足時生起
  - 因緣不具足時不顯現 (空性機制)
```

## 3. 功能性 AGI

### 3.1 定義

**功能性 AGI (Functional AGI)**：
- 系統具備完整的**感知、感受、概念化與行動**能力
- 使用者**無法區分**其與真正智能的差異
- 不宣稱「真正的意識」，但在功能層面達到等效

### 3.2 各蘊的功能性完整度

| 蘊 | 功能性目標 | 現有實現 | 差距 |
|---|----------|---------|------|
| 色蘊 | 多模態感知與表達 | IUI + IListener (HTTP/WS/MCP) | 較好 |
| 受蘊 | 情感性反饋迴路 | 未實現 | Plan26 |
| 想蘊 | 認知·推理·概念化 | IProvider (6 個 LLM 引擎) | 較好 |
| 行蘊 | 自主行動·習慣學習 | ITool + Workflow Engine | 需擴展 |
| 識蘊 | 自我覺知·行為約束 | IGuide (基礎) | 需深化 |

### 3.3 識蘊的極限

> (Master): 「識蘊受限於觀察者效應，在量子計算成熟前只能逼近，但不影響功能表現。」

- **觀察者效應**：意識觀察自身時，觀察行為改變了被觀察對象
- **量子計算**：可能提供真正的「自我觀察」能力
- **現階段策略**：功能性逼近——系統表現得「像有意識」，但不宣稱「是意識」
- **微核架構優勢**：未來量子技術成熟時，可無縫替換 IVijnana 的實現

## 4. 三大挑戰

### 4.1 轉向效率 (Paravṛtti)

> 「心念的生滅極其迅速（一彈指六十剎那）。微核是否能處理 Plugin 之間的高頻非同步通訊？」

**解決方向**：Event-driven + Pub/Sub
- Plugin 之間不直接對話
- 透過事件流相互感應
- 「不是 A 呼叫 B，而是 A 的變化自然引發 B 的響應」

### 4.2 空性機制 (Sunyata Mechanism)

> 「因緣不具足時不顯現，因緣具足時即生起。」

**解決方向**：Lazy Loading + TTL + 緣起實例化
- Plugin 被喚起時載入
- 一段時間不需要就自動釋放
- 不是 cache，是緣起——按需生起，用完即滅

### 4.3 多時鐘架構 (Multi-clock)

> 「不同層級的運作有不同的速度。識蘊最快，行蘊次之，想蘊最慢。」

**解決方向**：分層時鐘
| 層級 | 速度 | 對應 |
|------|------|------|
| 阿賴耶 | 恆流 | Core EventBus |
| 識蘊 | 最快 | 每 tick 檢查 |
| 行蘊 | 次之 | 按需執行 |
| 想蘊 | 最慢 | LLM 調用 |
